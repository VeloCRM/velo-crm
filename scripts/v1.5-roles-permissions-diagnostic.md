## 2. Proposed RLS / schema changes (specific SQL)

### 2a. Enum ŌĆö `ALTER TYPE` (Phase 1, standalone migration)
Live type is the **enum** `profile_role` (`schema.sql:38`; `profiles.role profile_role NOT NULL DEFAULT 'assistant'` at `:192`; enforced by `enforce_profile_immutable_fields` and the prescriptions doctor-role trigger). The text-CHECK redefinition in `scripts/multi-doctor-migration.sql:14ŌĆō22` is **dead legacy** (not applied ŌĆö confirmed in the V1.5 architecture diagnostic).

```sql
ALTER TYPE public.profile_role ADD VALUE IF NOT EXISTS 'xray_tech';
```
**Caveats (drive the migration ordering):**
- **Cannot be used in the same transaction it's added** (all PG versions) ŌåÆ the policy rewrites in 2b MUST be a separate, later migration/commit.
- **Forward-only** ŌĆö Postgres has no `DROP VALUE`; rollback = leave it unused (harmless) or recreate the type (heavy). Same constraint as the fracture/wear enum work.
- Run as its own statement; if the migration runner wraps in a tx, isolate this one.
- **Pre-flight (cheap insurance):** `SELECT pg_typeof(role) FROM public.profiles LIMIT 1;` ŌĆö expect `profile_role`. If it returns `text`, the live column is the legacy text variant and the change is `ALTER TABLE ... DROP/ADD CONSTRAINT profiles_role_check` instead (and the `role IN (...)` predicates already accept text).

### 2b. xrays RLS ŌĆö add `xray_tech` to write policies (Phase 2, after enum commit)
7 sites, each `role IN ('owner','doctor')` ŌåÆ `role IN ('owner','doctor','xray_tech')` (drop+recreate or `ALTER POLICY`):
- **Table** (`scripts/xray-module-migration.sql`): INSERT `:113ŌĆō118`, UPDATE `:121ŌĆō134`, DELETE `:136ŌĆō143`
- **Storage** (`patient-xrays` bucket, same file): INSERT `:176ŌĆō186`, UPDATE `:190ŌĆō200`, DELETE `:204ŌĆō214`
- **Hotfix** (`scripts/xray-fix-storage-insert-policy.sql:32`) ŌĆö update so a fresh apply stays consistent

xrays SELECT is org-only (`xrays_select_own_org`, `:108ŌĆō109`) ŌåÆ `xray_tech` already reads. No change needed for read.

### 2c. Clinical RLS ŌĆö LEAVE AS-IS for V1.5 (documented limitation)
All core clinical tables (`patients`, `dental_chart_entries`, `treatment_plans`, `prescriptions`, `notes`, `payments`, `appointments`, ŌĆ”) are **org-only RLS, no role dimension** (`schema.sql:727ŌĆō1195`). `'receptionist'` appears in SQL only in the documents *storage* policy (`patient-documents-bucket.sql:76,95,114`).

> **Add to `scripts/security-audit-2026-06-24.md`:** *"V1.5 receptionist clinical read-only is **UI-enforced only**; clinical-table RLS remains org-scoped (any authenticated org member can read/write clinical rows at the DB). V2 adds DB-enforced PHI separation (the `patient_clinical` split + role-aware policies per `scripts/v1.5-architecture-diagnostic.md` ┬¦3/┬¦6). Until then, a malicious receptionist with API access could write clinical data the UI hides."*
