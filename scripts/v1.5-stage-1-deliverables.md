## ⚠️ Flags raised in Stage 1 (decide before Stage 2)

- **Payments are likely NOT structured in GHL** — the prior import regex-scraped them from note prose. The exploration probe confirms. If confirmed, this is a **blocking product decision** (regex / Saif spreadsheet / fresh start). *(BLOCKING)*
- **IQD 1000× risk** — `*_minor` columns hold whole dinars (D-1); GHL prose used `"X 000id"`=×1000. Lock the multiplier before any payment import. *(BLOCKING)*
- **`patients` has no external-import key** — re-import idempotency relies on phone uniqueness only; `notes`/`documents` already have `external_id`. Recommend adding `external_id`+`external_source` to `patients` (and a dedupe key to `payments`) as the first Stage-2 migration. *(HIGH)*
- **Document signed URLs expire** — must download-during-import to `patient-documents` bucket (not the legacy `documents` bucket); mind the 8-MIME / 25 MB limits. *(HIGH)*
- **Custom fields may hide clinical data** — enumerate every key in the sample; a clinical custom field changes Stage-2 scope. *(HIGH — verify)*
- **Legacy import scripts target the DELETED schema** — `scripts/velo-import.mjs` / `extract-payments.mjs` / `fetch-documents.mjs` write to `contacts`/old `payments`/old `documents` columns and the `documents` bucket. They are the **SB-1 source** and must be REPLACED in Stage 2 (kept only as reference for GHL API patterns; do not run them against the current schema).
