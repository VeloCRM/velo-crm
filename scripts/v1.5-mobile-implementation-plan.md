**Key architectural call:** do the iPhone/iPad split with **pure Tailwind dual-render visibility** (`hidden md:block` for the wedge arch, `md:hidden` for the compact grid) — **no new `useMediaQuery` hook, no prop-drilling `isMobile`**. Avoids new JS infra, works with resize/SSR automatically, satisfies "Tailwind sufficient." (Fallback: a tiny `useIsMobile` hook if dual-render proves heavy — but the SVG arch is cheap and the hidden tree costs no layout.)

## M-01 — Dental chart hybrid redesign (detailed design)

**Who it affects:** owner/doctor (edit) + receptionist/assistant (read) — the roles that see the chart tab. `xray_tech` has the tab hidden (PR #50), so no change for them. `canEdit` (`DentalTabs.jsx:387-389`, `EDIT_ROLES={owner,doctor}`) already gates editing; the redesign reuses it unchanged.

### Backbone (verified)
- `ToothSurfaces({fdi, findings, findingStyles, onSurfaceClick, onAddClick, notation, locale, disabled})` — `onSurfaceClick(surface)` emits the **DB surface value** (`mesial/distal/buccal/lingual/occlusal`, never `incisal`); `onAddClick()` (no arg) is the whole-tooth entry. SVG capped at inline `style={{maxWidth:46}}` (`ToothSurfaces.jsx:64`).
- `DentalChartTab` arches: two `grid-cols-16 gap-1.5` containers **hard-pinned `dir="ltr"`** (`DentalTabs.jsx:568,572`); `renderTooth(n)` wires `onSurfaceClick={s=>openSurface(n,s)}`, `onAddClick={()=>openTooth(n)}`, `disabled={!canEdit}` (`:516-528`).
- **The add-finding modal is fully driven by 3 state vars** — `activeTooth`, `prefillSurface`, `form` — set by `openTooth(n)` (`:428-434`) / `openSurface(n,surface)` (`:437-443`). **Reuse contract: a mobile UI just calls these handlers; the existing modal + `addDentalChartEntry` submit path are untouched.**
- Labels: `SURFACE_LABELS` (en/ar, `toothSurfaces.js:24-31`); `occlusal` shows as **Incisal/قاطعة** for anterior teeth (`fdi%10 ≤ 3`, see `surfaceOptionLabel` `:450-453`). `fracture`/`wear` require a real surface (`SURFACE_REQUIRED_FINDINGS`); whole-tooth findings (`WHOLE_TOOTH_FINDINGS`) force surface=null.

### iPad path (≥768px) — enlarge, minimal
- Replace the inline `style={{maxWidth:46}}` (`ToothSurfaces.jsx:64`) with responsive Tailwind on the `<svg>`: `className="w-full h-auto max-w-[46px] md:max-w-[60px] lg:max-w-[72px]"` (drop the inline cap — it can't be made responsive). On iPad landscape (`lg`, 1024px) a tooth becomes ~64–72px → center wedge ~19–22px; portrait (`md`) ~60px. Still <44px, but a large jump from ~14px and workable with a precise tap/stylus chairside (the locked decision accepts this for iPad).
- **M-02** add `active:[stroke:#06b6d4] active:[stroke-width:2.5]` to `WEDGE_CLASS` (`ToothSurfaces.jsx:31`) → touch press feedback (hover never fires on touch).
- **M-03** give the tooth-number whole-tooth label (`ToothSurfaces.jsx:117-123`) a real hit area on touch: `min-h-[44px] inline-flex items-center justify-center` (or fold into the sheet on iPhone).
- Effort: **XS** (~10 lines, pure Tailwind).

### iPhone path (<768px) — tap-to-sheet (the L item)
- In `DentalChartTab`, **dual-render**: wrap the existing two wedge arches in `hidden md:block`; add a new compact arch `md:hidden`.
- **Compact arch:** same `dir="ltr"` 16-col grid, but each tooth is a single **≥44px tooth button** (number via `<ToothLabel>` + a small finding-color dot derived from `groupBySurface(entries[tooth])` so the user sees which teeth have findings). Tapping sets local `sheetTooth` state.
- **New `src/components/MobileToothSheet.jsx`** (Open Q2 — lives in `components/`, sibling to `ToothSurfaces`/`ToothLabel`):
  - Renders inside the shared `Modal` (`shared.jsx`, already bottom-sheets on mobile, `max-h-85vh`).
  - Header: large `<ToothLabel fdi notation locale>` (~big) + close ×.
  - Body: **6 full-width ≥44px rows** — Mesial / Distal / Buccal / Lingual / Occlusal(→Incisal label if anterior) / **Whole tooth** — each showing the current finding on that surface (color chip + `SURFACE_LABELS`/`findingLabel`, EN+AR) and, when `canEdit`, acting as the tap target.
  - **Tap a surface row → close the sheet AND call `onSurfaceClick(surfaceDbValue)`** (= `openSurface(sheetTooth, surface)`); the Whole-tooth row → `onAddClick()` (= `openTooth`). The existing add-finding modal then opens. **Closing the sheet before opening the modal avoids two stacked bottom-sheets** (critical gotcha — see below).
  - Read-only roles: rows render finding state but are non-interactive (`canEdit=false`).
- Reuse: `MobileToothSheet` takes the **same prop contract as `ToothSurfaces`** (`fdi, findings, findingStyles, onSurfaceClick, onAddClick, notation, locale, disabled`), so `DentalChartTab` wires it identically. No new modal, no new submit path, no schema/data change.
- Effort: **L** (~180–230 LOC: new component + compact grid + wiring).

### RTL
Arch + sheet tooth geometry stay **LTR/anatomical** (never mirror — comments at `ToothSurfaces.jsx:18`, `DentalTabs.jsx:566`). Only text/labels follow `dir`. Surface labels already EN/AR.

### Guardrail check (M-01 LOC)
iPad (~10 LOC) + iPhone (~200 LOC) ≈ **~210 LOC < 300** → fits one PR (PR A). **If it exceeds 300 during build, split:** ship iPad-enlarge + M-02/M-03/M-04 in PR A, move the iPhone `MobileToothSheet` to its own PR A2.

## Implementation gotchas

- **Two stacked bottom-sheets:** the shared `Modal` bottom-sheets on mobile; the add-finding modal is also a `Modal`. **Close `MobileToothSheet` before calling `openSurface`/`openTooth`** so only one sheet shows. (Designed in above.)
- **Inline styles block Tailwind responsiveness** — several fixes require converting inline `style` → classes/props, not just adding variants: `ToothSurfaces` `maxWidth:46` (M-01), `AddAppointmentModal` `gridTemplateColumns` (M-17), plan-table `height/fontSize` (M-11). Can't be done with `md:` alone.
- **Dual-render cost:** both arches (wedge + compact) render; the hidden one is `display:none` (no layout cost). 32 extra tooth buttons is trivial vs the 160 SVG polygons already drawn.
- **`max-sm:`/`max-md:` are first use in this repo** (Tailwind v4 supports them; zero current usages) — verify they compile in the build before relying on them broadly; `md:`-prefix (mobile-first) is the safer default where equivalent.
- **`button{min-height:44px}` (M-14)** is inside the ≤768 media query → desktop unaffected; but re-check dense mobile button rows don't overflow after the bump.
- **xray_tech unaffected** — chart tab hidden (PR #50); no role logic changes in this pass.
- **Per the "Success ≠ applied" lesson** — these are code/CSS changes (deploy-verified), not SQL; standard build + device test, no pg_policies concern.
