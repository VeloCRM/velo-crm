# V1.5 Architecture Diagnostic — Role-Scoped Patient Ownership

**Phase 1, read-only. No code, no schema, no migrations.** Date: 2026-06-24.
**Goal:** complete architectural plan for V1.5 — role-scoped patient ownership (owner / doctor / receptionist / new xray_tech), doctor-to-doctor patient sharing, and Le Royal backfill — ahead of Saif Dental onboarding (mid-to-late July).

Evidence base: full read of `src/lib/schema.sql` + per-PR migration scripts, all `src/lib/*.js` data-layer helpers, all 12 `api/**` endpoints, and the frontend role-gating surface. File:line citations throughout.

---

## Executive summary

**Estimate: ~7 PRs across ~2 weeks of architecture work (Weeks 1–2 of the V1.5 plan), then UI + GHL import.** This is **multiple sequenced deployments, not one PR** — additive schema → backfill → RLS cutover → app/UI → API hardening.

**The five findings that shape everything:**

1. **Current clinical RLS is org-only — zero role/doctor dimension.** Every clinical table's policies are `org_id = current_org_id()` for all four ops (the lone exception: `xrays` gates writes to owner/doctor). The `permissions.js` MATRIX is **client-side UX only** — the DB lets any org member read/write every patient. V1.5 is the first time access control moves into the database. (`schema.sql:729–1195`; `permissions.js:5–7`.)

2. **`patients.primary_doctor_id` already exists** (PR #6) as a *filter-only, non-security* column (1:1, `ON DELETE SET NULL`, indexed, **not referenced by any RLS policy**). **Recommendation: promote this existing column to the assignment boundary — do NOT add a parallel `assigned_doctor_id`** (two sources of truth = drift). The prompt's `assigned_doctor_id` is satisfied by `primary_doctor_id`.

3. **Receptionist "demographics-only" is NOT achievable with row-level RLS.** `patients` mixes demographics (`full_name`, `phone`, `dob`) with clinical PHI (`medical_history`, `allergies`) in one row, and **column-level GRANTs can't distinguish app-roles** (every clinic user is the same Postgres `authenticated` role). The only DB-enforced fix is to **split clinical PHI out of `patients`** into a clinical-tier table. This is the single largest schema change in V1.5. (`schema.sql:202–219`; `dental.js:144–225`.)

4. **🔴 Service-role bypass: `api/whatsapp/send.js` is user-facing, has no role check, and reads/writes `patients`/`conversations`/`messages` under the service-role key** — RLS will never see this traffic. Any role can pass any in-org `patientId`, confirm the patient + read their phone, and send/persist messages. **This path must replicate role/ownership checks in application code, or V1.5 RLS has a hole.** (`api/whatsapp/send.js:27,61,129`.)

5. **The patient profile has no read-side role gating today** — every role sees all 10 clinical tabs; only *writes* are disabled. Hiding clinical detail from receptionist/xray_tech is net-new gating, and `PatientProfile` doesn't even receive the current role. (`App.jsx:1747–2040`.)

**Recommended access model:** keep `primary_doctor_id` as the primary assignment, add a `patient_sharing_grants` table for additive doctor-to-doctor shares (no handoff, **no transitive re-sharing**), and have all child clinical tables **inherit access via `patient_id` through one shared `can_access_patient()` SQL helper** rather than denormalizing a doctor column onto each (no reassignment triggers, single source of truth).

**Top risks:** (a) the RLS cutover changes result sets for every clinical query — org-wide aggregates (dashboard/reports/finance) silently shrink for scoped doctors; (b) the whatsapp/send bypass; (c) backfill ordering — doctors lose visibility if policies swap before `primary_doctor_id` is populated; (d) 12/12 Vercel functions means no new endpoints (all API fixes in-place, RLS is free).

## 3. Assignment + inheritance model (DECISION)

### Assignment column: REUSE `primary_doctor_id`
- It already exists, is indexed (`patients_primary_doctor_idx (org_id, primary_doctor_id)`), and the UI + data layer already read/write it (the "My patients" toggle, `fetchPatients({primaryDoctorId})`, `getMyPatientsCount`, the patient-form doctor picker). Promoting it from filter to RLS boundary reuses all of that.
- Adding a second `assigned_doctor_id` would create two sources of truth and silently diverge from the PR #6 machinery. **Rejected.**
- Caveat to carry: `ON DELETE SET NULL` means deleting a doctor un-assigns their patients (they fall back to owner-only visibility) — acceptable and safe; note it in runbooks.

### Inheritance: child tables inherit via `patient_id`, NOT denormalized doctor columns (DECISION)
Two options were weighed:

- **(A) Denormalize `assigned_doctor_id` onto each clinical table** — fast (no JOIN) but requires UPDATE triggers on every table to stay consistent on reassignment *and* on every share grant/revoke (a share isn't a single column anyway). N tables × trigger maintenance, drift risk, and exactly the "Supabase `.update()` silently no-ops under RLS" failure class we've already been bitten by.
- **(B) Inherit access through `patient_id` via one shared predicate** — every child policy calls `can_access_patient(patient_id)`; grandchildren (`treatment_plan_items`, `prescription_items`, `messages`) resolve through their parent's `patient_id`/`conversation_id`. Single source of truth (assignment + shares live in two places only: `patients.primary_doctor_id` and `patient_sharing_grants`). No reassignment triggers.

**Recommendation: (B).** It's the altitude-correct choice — generalize the access rule into one SQL function and reuse it, rather than special-casing a denormalized column on 12 tables. Performance is mitigated by making the helper `STABLE` (planner caches per statement) and indexing the two lookup paths (see §13). At Saif's scale (~3,171 patients, modest rows per table) the JOIN cost is negligible.

## 4. `patient_sharing_grants` table (DESIGN)

```
patient_sharing_grants
  id           uuid PK default gen_random_uuid()
  org_id       uuid NOT NULL → orgs(id) ON DELETE CASCADE        -- tenancy axis (matches every table)
  patient_id   uuid NOT NULL → patients(id) ON DELETE CASCADE    -- grant dies with patient
  doctor_id    uuid NOT NULL → profiles(id) ON DELETE CASCADE    -- grant dies if doctor removed from org
  granted_by   uuid → profiles(id) ON DELETE SET NULL            -- who shared (audit)
  granted_at   timestamptz NOT NULL default now()
  revoked_at   timestamptz                                       -- NULL = active (soft delete)
  revoked_by   uuid → profiles(id) ON DELETE SET NULL
-- one ACTIVE grant per (patient, doctor):
CREATE UNIQUE INDEX ... ON patient_sharing_grants (patient_id, doctor_id) WHERE revoked_at IS NULL;
-- access lookup path:
CREATE INDEX ... ON patient_sharing_grants (doctor_id, patient_id) WHERE revoked_at IS NULL;
```

**RLS:**
- SELECT: org members (or scope to owner + involved doctors) + operator.
- INSERT (grant): **owner OR the patient's `primary_doctor_id`** — and `WITH CHECK` that `doctor_id` is a `role='doctor'` profile in the same org (mirror the `enforce_prescription_doctor_role` pattern).
- UPDATE (revoke = set `revoked_at`): owner OR `granted_by` OR the patient's primary doctor.
- No hard DELETE for clinic users (revoke is a soft update); operator bypass.

**Cascade behavior:** patient deletion → grants cascade away; doctor removed from org (profile delete) → their grants cascade away (they lose shared access automatically). Both correct.

> ⚠️ **GUARDRAIL — transitive sharing (MUST CONFIRM):** The INSERT rule "only the **primary** doctor or owner may grant" **prevents transitive re-sharing** — a doctor who *received* a share (Hawkar) cannot re-share to a third doctor. This matches referral semantics (Saif → Hawkar, not Saif → Hawkar → X) and keeps the access graph auditable and shallow. **Open question for Ali: confirm transitive sharing is disallowed** (recommended). If it must be allowed, the INSERT predicate widens to "any doctor with active access," and the cascade/audit story gets materially more complex.

> **No handoff:** sharing is purely additive — `primary_doctor_id` is untouched by a grant, so the primary doctor never loses access. Matches the spec ("both doctors see shared patient, no losing access").

## 6. Proposed RLS (pseudo-SQL)

**New helpers (add first):**
```sql
-- role of the caller (mirror of current_org_id)
CREATE FUNCTION current_user_role() RETURNS profile_role
  LANGUAGE sql STABLE SECURITY DEFINER SET search_path=public AS
$$ SELECT role FROM profiles WHERE id = auth.uid() $$;

-- single source of truth for "may this caller access this patient's clinical data"
CREATE FUNCTION can_access_patient(p_patient uuid) RETURNS boolean
  LANGUAGE sql STABLE SECURITY DEFINER SET search_path=public AS
$$
  SELECT is_operator() OR EXISTS (
    SELECT 1 FROM patients pt
    WHERE pt.id = p_patient
      AND pt.org_id = current_org_id()
      AND (
        current_user_role() = 'owner'
        OR (current_user_role() = 'doctor' AND (
              pt.primary_doctor_id = auth.uid()
              OR EXISTS (SELECT 1 FROM patient_sharing_grants g
                         WHERE g.patient_id = pt.id AND g.doctor_id = auth.uid()
                           AND g.revoked_at IS NULL)
        ))
      )
  );
$$;
```
`can_access_patient` deliberately returns **false for receptionist/xray_tech** — those roles do NOT get clinical row access; they're handled per-table below (demographics / x-ray-only).

**Per-table policies (USING / WITH CHECK shown conceptually):**

**patients (demographics tier — after the PHI split in §3 caveat / §11):**
- SELECT: `org_id = current_org_id()` for **owner, receptionist, xray_tech** (all org patients) **OR** `can_access_patient(id)` for **doctor** (assigned+shared). Operator bypass.
- INSERT: owner, receptionist (front desk registers patients), doctor (auto-sets `primary_doctor_id = self`). xray_tech denied.
- UPDATE: owner/receptionist demographics; doctor only for assigned+shared. (PHI columns live in the clinical table below, so receptionist physically cannot touch them.)
- DELETE: owner only.

**patient_clinical (NEW — `medical_history`, `allergies`, `patient_id` PK, `org_id`):** see §11.
- SELECT/INSERT/UPDATE/DELETE: `can_access_patient(patient_id)` (owner + assigned/shared doctor only). Receptionist + xray_tech have **no policy → no access**.

**dental_chart_entries, treatment_plans, treatment_plan_items, prescriptions, prescription_items, notes, form_submissions, conversations, messages (clinical):**
- SELECT: `can_access_patient(<patient_id resolved directly or via parent>)`. (treatment_plan_items → join `treatment_plans`; prescription_items → join `prescriptions`; messages → join `conversations`.)
- INSERT/UPDATE/DELETE: `can_access_patient(...)` **AND** `current_user_role() IN ('owner','doctor')`. Receptionist/xray_tech denied.

**xrays (special — xray_tech is org-wide):**
- SELECT: `org_id = current_org_id()` for **owner, xray_tech** (all) **OR** `can_access_patient(patient_id)` for **doctor**. Receptionist denied (clinical image).
- INSERT/UPDATE/DELETE: `current_user_role() IN ('owner','xray_tech')` (any patient in org) **OR** (`doctor` AND `can_access_patient(patient_id)`). This is the one place the existing role-gated xray policy is extended rather than rewritten.

**appointments (front desk needs all):**
- SELECT: owner/receptionist/xray_tech → org-wide; doctor → `can_access_patient(patient_id)`.
- INSERT/UPDATE/DELETE: owner/receptionist (front desk books) + doctor for assigned+shared. xray_tech read-only.

**payments (finance — front desk writes, doctor reads):**
- SELECT: owner/receptionist → org-wide; doctor → `can_access_patient(patient_id)` (read); xray_tech denied.
- INSERT/UPDATE/DELETE: owner/receptionist only.

**documents:** treat as clinical (recommended) → same as dental tables (owner + assigned/shared doctor; receptionist/xray_tech denied). **Open question (§ open questions): is Documents "clinical" or front-desk paperwork?** Today receptionist *can* edit documents (`DOCUMENTS_EDIT_ROLES`), so confirm before hiding.

All tables keep the existing `is_operator()` bypass policy unchanged.

**Intent:** every existing Le Royal patient assigned to `alialjobory89` (owner) on V1.5 deploy; owner reassigns later.

**Order:** backfill **before** the RLS cutover — otherwise doctors (none scoped yet) and the owner are fine, but the moment policies flip, any NULL-assignment patient becomes owner-only. Since everything is assigned to the owner and the owner sees all, there's no visibility gap during transition.

```sql
-- resolve owner profile id for Le Royal, then:
UPDATE patients SET primary_doctor_id = :owner_id
 WHERE org_id = '66f75f33-40b0-4f00-bf33-b9ac1c20af46'
   AND primary_doctor_id IS NULL;     -- see edge case below
```

> ⚠️ **GUARDRAIL — backfill edge cases:**
> - **Existing PR #6 assignments:** PR #6's "My patients" toggle may already have set `primary_doctor_id` on some Le Royal patients. The spec says "all assigned to owner," but **force-overwriting would destroy real doctor assignments.** Recommend `WHERE primary_doctor_id IS NULL` (preserve existing). **Open question: force-all or NULL-only?**
> - **Owner is not role='doctor':** assigning patients to an owner is fine for visibility (owner sees all regardless), but those patients won't appear for Saif/Hawkar until reassigned — expected, owner reassigns post-import.
> - **Legacy/GHL rows with no `created_by`:** irrelevant to assignment (we key on `primary_doctor_id`), covered by the NULL backfill.
> - **PHI split backfill:** the `medical_history`/`allergies` → `patient_clinical` migration must copy every existing row (including empty `{}`/`[]` defaults) so no patient loses clinical data.

**Verification (before/after):**
```sql
-- before: how many unassigned?
SELECT count(*) FILTER (WHERE primary_doctor_id IS NULL) AS unassigned, count(*) AS total
  FROM patients WHERE org_id = :leroyal;
-- after: distribution by assigned doctor

     = (SELECT count(*) FROM patient_clinical c JOIN patients p ON p.id=c.patient_id WHERE p.org_id=:leroyal) AS parity_ok;
```

**Rollback:** assignment backfill is reversible (`SET primary_doctor_id = NULL`). The risky step is the policy cutover — keep the prior org-only policy DDL in a `rollback-rls.sql` so re-applying it restores V1 behavior instantly; the new columns/table/helpers are additive and can remain.

## 13. Risk assessment

- **Performance:** `can_access_patient()` runs per row on clinical queries. Mitigations: `STABLE` helper (planner caches within a statement), partial indexes on `patient_sharing_grants (doctor_id, patient_id) WHERE revoked_at IS NULL` and the existing `patients (org_id, primary_doctor_id)`; child-table EXISTS resolves on `patients` PK. At Saif's scale (3,171 patients, small per-table row counts) impact is negligible — **benchmark the heaviest clinical list query on a Le Royal-sized copy before the Le Royal cutover.**
- **Correctness:** every clinical query's result set changes; org-wide aggregates (dashboard/reports/finance) silently shrink for scoped doctors. Each §7 aggregate needs an explicit caseload-vs-org decision and a test. This is the biggest review burden.
- **Backward compat:** Le Royal today operates effectively as owner-only → owner sees all, so the transition is invisible to current users *provided* backfill precedes cutover. Doctors/receptionist/xray_tech are net-new.
- **The whatsapp/send bypass** must ship in the cutover cycle or RLS is incomplete.
- **Rollback:** keep prior policy DDL; cutover is reversible by re-applying old policies. Additive schema (columns/table/helpers/enum value) stays — note that **an enum value cannot be removed**, so `xray_tech` is permanent once added (acceptable, additive).
